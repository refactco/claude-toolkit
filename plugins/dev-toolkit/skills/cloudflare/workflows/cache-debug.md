# Workflow: Cache Debug

Diagnose why Cloudflare isn't caching as expected and write Cache Rules to fix it.

**When to use:** cache hit rate is low, `Cf-Cache-Status` comes back `DYNAMIC` / `BYPASS` on pages that should HIT, or origin CPU is high from cacheable pages.

**Do NOT use for:** slow-origin problems unrelated to cache, or WAF issues (waf-rules workflow).

**Conventions (see `../SKILL.md`):** Narration · MCP servers (`cloudflare-api`, `cloudflare-docs`, `cloudflare-graphql`) · Resolve the zone · Role check.

## Step 1 — Capture headers

Don't trust descriptions; pull headers. Hit `https://{{client_domain}}/{{path}}` three times back-to-back (cache-busting headers off) to reveal MISS→HIT progression vs persistent MISS.

Headers to capture:
`cf-cache-status`, `cf-ray`, `cache-control`, `age`, `vary`, `set-cookie`, `x-wpengine`, `x-wpengine-cache`.

## Step 2 — Decode Cf-Cache-Status

| Value | Meaning | Path |
|---|---|---|
| `HIT` | Cached at edge | Done; check `age:` is reasonable |
| `MISS` → `HIT` | Normal first-visitor warm | Done |
| `MISS` forever | Cache-key fragmentation (cookies / query strings) | Step 3d |
| `BYPASS` | Origin sent `no-cache` / `private` / `max-age=0` | Step 3a |
| `DYNAMIC` | Cloudflare default-skips this (HTML on Free/Pro/Biz) | Step 3b |
| `NONE`/`UNKNOWN` | Cloudflare itself responded (WAF / Worker / redirect) | Step 3c |

See `references/wordpress/cache-status-values.md` for the full table.

## Step 3 — Causes

### 3a. BYPASS — origin says no-cache

Look at `cache-control:`. Publisher culprits:
- `no-cache, no-store, private` — popup / analytics / membership plugin sees a tracking cookie and assumes "logged in".
- `Cache-Control: max-age=600, public` — WP Engine default for HTML. Override via Cache Rule (see 3b).

Fix: Cache Rule with `edge_ttl.mode: "override_origin"`. The override is mandatory; without it, the rule still respects the origin's `private`.

See `references/wordpress/wp-engine-cache.md` for WP Engine's specific behavior.

### 3b. DYNAMIC — Cloudflare didn't cache by default

On Free/Pro/Biz, HTML isn't cached by default. Create a Cache Rule in the `http_request_cache_settings` phase with action `set_cache_settings`.

Required `action_parameters`:
- `cache: true`
- `edge_ttl.mode: "override_origin"` — without it, origin's `private` still wins
- `edge_ttl.default` — TTL guidance: news 600–1800, magazine 3600–14400, marketing 86400+
- `edge_ttl.status_code_ttl: [{ status_code: 200, value: 14400 }, { status_code: 404, value: 600 }]`
- `browser_ttl.mode: "override_origin"`, `browser_ttl.default: 3600`
- `cache_key.ignore_query_strings_order: true`, `cache_by_device_type: false`
- `respect_strong_etags: true`

Expression (host + exclude admin / logged-in cookie):

```
(http.host eq "{{client_domain}}" and not (http.request.uri.path matches "^/wp-(admin|login)" or http.cookie contains "wordpress_logged_in_"))
```

Two things this rule MUST have:
- `edge_ttl.mode: "override_origin"` — without it, origin's `private` still wins.
- Exclude `wordpress_logged_in_*` cookie — without it, editors see each other's cached pages.

See `references/wordpress/wordpress-cookies.md` for the full cookie picture.

### 3c. NONE / UNKNOWN

Response generated by CF itself. Query firewall + access events via the GraphQL MCP to find the actor.

### 3d. MISS forever — cache-key fragmentation

UTM / HubSpot params or per-visitor cookies make every key unique. Cache Rule with `cache_key.custom_key.query_string.exclude.list` set to the tracking params:

```
__hstc, __hssc, __hsfp, _ga, fbclid, gclid,
utm_source, utm_medium, utm_campaign, utm_term, utm_content
```

Also set `cache_key.ignore_query_strings_order: true` and `edge_ttl.mode: "override_origin"`. Use the same logged-in-cookie exclusion in the expression.

See `references/wordpress/tracking-params.md` for the full list of params to strip.

## Step 4 — Verify

Re-pull headers (same loop as Step 1): expect MISS → HIT, HIT with UTM, BYPASS/DYNAMIC when the request carries `wordpress_logged_in_`.

For a 24h hit-rate breakdown, run a GraphQL query against `httpRequestsAdaptiveGroups`:
- Filter: `datetime_geq` = 24h ago, `edgeResponseContentTypeName_like: "%html%"`.
- Group by `dimensions { cacheStatus }`, order by `count_DESC`.

Target HIT share: 80%+ content-heavy, 95%+ mostly-static.
